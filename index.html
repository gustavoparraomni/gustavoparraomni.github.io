CASO 01 · LO MÁS BÁSICO

Un feature, de principio a fin

El recorrido completo del feature AhorroMarket: nace de develop, se desarrolla en su rama propia, y vuelve a integrarse con un merge commit identificable.

→
El merge commit es lo importante El círculo grande en develop representa un único punto de entrada. Los 4 commits del feature están agrupados bajo él. Si hay que revertir, es un solo comando: git revert -m 1 <hash>.
1
Crear rama desde develop
El feature siempre nace de develop, nunca de otra rama de feature.
git checkout -b feature/AhorroMarket
2
Trabajar libremente
La rama no está protegida. El owner mergea sub-ramas a su discreción.
commit, commit, commit...
3
PR a develop con --no-ff
Genera el merge commit identificable que será el ancla del feature.
git merge --no-ff feature/AhorroMarket
CASO 02 · DOS EQUIPOS A LA VEZ

Dos features, dos equipos, una develop

El equipo A construye AhorroMarket mientras el equipo B construye PagosMercado. Ambas ramas viven en paralelo, sincronizando con develop frecuentemente para evitar conflictos al final.

!
Sync diario, no semanal Las líneas punteadas muestran cuando cada feature trae lo último de develop. Hacerlo seguido evita el "merge hell" al final. Si la rama vive 3 semanas sin sync, los conflictos finales son inmanejables.
A
Ramas independientes
Cada equipo trabaja en su rama sin pisar al otro. Si tocan archivos compartidos, lo coordinan por canal del equipo.
B
Sincronización constante
Cada 1-2 días: git merge develop dentro de la rama del feature para resolver conflictos pequeños temprano.
CASO 03 · RELEASE BRANCH PARA QA

Cuando los features están listos para QA

Se crea una rama release-stg/9.22.0 desde develop. Ahí se generan los builds para QA, se aplican los fixes que ellos detecten, y cuando aprueban se promueve a staging.

i
Numeración de builds Cada build subido a tienda incrementa el número entre paréntesis: v9.22.0(1), v9.22.0(2)... El que QA aprueba es el que se promueve. Esto cumple el requisito de iOS y Android de tener un build number único por upload.
① rama desde develop ② build (1) → QA ③ fix encontrado ④ build (2), (3)... iterando ⑤ aprobado → staging ⑥ staging → develop (sync)
CASO 04 · CICLO DE PRODUCCIÓN

Camino a las tiendas y merge en reversa

Una vez staging tiene la versión, se crea release/9.22.0 para el ciclo productivo. Si aparecen bugs solo visibles en producción, se arreglan ahí. Cuando sale a tiendas, hacemos merge en reversa.

↺
El merge en reversa es OBLIGATORIO Después de la salida a producción: production → staging → develop. Si lo olvidamos, los fixes que solo aparecieron en producción se reintroducen como bugs en la próxima versión.
# Después de que la versión sale a tiendas $ git checkout production $ git merge --no-ff release/9.22.0 -m "PROD v9.22.0" $ git tag v9.22.0 # Merge en reversa — propagamos hacia atrás $ git checkout staging && git merge production $ git checkout develop && git merge staging
CASO 05 · EMERGENCIA EN TIENDAS

Bug crítico en producción → Hotfix

Un crash crítico aparece en la versión v9.22.0 que ya está en tiendas. No podemos esperar la próxima versión. Sale una rama directo de production, se arregla, se publica como v9.22.1 y se propaga hacia atrás.

!
El hotfix sale de production, no de develop Es la única rama que rompe la regla "todo nace de develop". Esto garantiza que el fix se aplica exactamente sobre el código que está corriendo en tiendas, sin contaminarse de features no liberados que existen en develop.
1
Sale de production
git checkout production
git checkout -b hotfix/9.22.1-T999
2
Builds incrementales
v9.22.1(1), v9.22.1(2)... hasta que QA aprueba el build correcto.
3
Merge a production + tag
git merge --no-ff hotfix/9.22.1-T999
git tag v9.22.1
4
Reversa obligatoria
production → staging → develop
Sin esto, el bug volvería en la próxima versión.
CASO 06 · EL CLIENTE CAMBIÓ DE OPINIÓN

Retirar un feature ya mergeado

El feature AhorroMarket ya está en develop, pero el cliente decide cancelarlo. Gracias al --no-ff y la convención de commits, podemos identificar el merge commit exacto y revertirlo en un comando.

⚠
Si el feature ya llegó a staging o production El revert se aplica también allí, en este orden: develop → staging → production. Si está en producción, técnicamente es un hotfix con propósito de revert.
1
Backup primero
Antes de tocar nada, una rama de respaldo por si el cliente cambia otra vez de opinión.
git checkout -b backup/develop-pre-revert
2
Identificar el merge commit
Buscamos por convención el commit "DEV AhorroMarket".
git log --merges --grep="AhorroMarket"
3
Revertir el merge
Un solo comando deshace todo el feature, manteniendo el historial limpio.
git revert -m 1 <hash>
i
¿Y si el feature vuelve después? No hagas merge directo de la rama original — git creerá que ya está integrado. El comando correcto es git revert <hash-del-revert> (revertir el revert).
CASO 07 · TODO JUNTO

El flujo completo en una sola vista

Todas las ramas, todos los ambientes, todos los movimientos juntos. Útil como póster para el equipo.

→
Flujo hacia adelante feature → develop → release-stg → staging → release → production
←
Flujo en reversa production → staging → develop (después de cada salida y cada hotfix)
★
Las 3 reglas que sostienen todo 1. Todo merge a rama protegida usa --no-ff  ·  2. Convención de commits: DEV / STG / PROD / fix(scope)  ·  3. Después de cada salida a producción, merge en reversa
CASO 08 · ESTADOS DE JIRA

Cómo se mueven las cards al ritmo de las ramas

El recorrido de un ticket desde que se planea hasta que QA está probándolo. Cinco estados, mapeados al evento de git que los dispara. La regla mental: si el código avanzó de ambiente, la card también.

# Estado Jira Trigger en git Quién mueve
01 To Do Ticket refinado y en el sprint, aún sin rama. PM / Tech Lead · manual
02 In Progress git checkout -b feature/AhorroMarket Dev · manual al empezar
03 Deploy Integration Dev terminó: PR enviado a la rama correspondiente. Dev al abrir el PR · manual
04 Deploy Staging PR aprobado y mergeado a la rama correspondiente. Reviewer al mergear · manual
05 QA Staging El desarrollo salió en una versión que QA ya está probando. Script de versionado · automático
⚙
Una transición es automática: QA Staging Cuando se genera una versión, un script revisa los tickets incluidos en la build y los pasa a QA Staging en bloque. Las demás transiciones las hace la persona responsable a mano.

Casos especiales

↺
QA encontró un bug
La card vuelve a In Progress. Cuando el fix se mergea de vuelta a la rama de release, el script la regresa a QA Staging en la siguiente build.
✕
Feature cancelado
La card se mueve a Won't Do. Si ya estaba mergeado, se sigue el flujo de cancelación con backup/* + git revert (ver Caso 06).
★
Por qué importa mantener Jira sincronizado Los reportes (sprint progress, release notes, métricas de ciclo) se construyen sobre los estados de Jira. Si las cards se quedan atrás del código, los dashboards mienten y nadie sabe qué está realmente listo para QA o ya en tiendas.
